# SPEC_WAVE3 · 第三波交接（多人发送 + 群发送）

> 前置：第二波已通过——Android 真机上能扫码登录、获取联系人、收发 1 对 1 文本消息。
> 本波在此基础上加两个小功能。仍在仓库 magicxiaomin/wa 工作。

---

## 0. 定位（重要，决定所有设计取舍）

这是一个**个人兴趣项目，只在 3-5 人小圈子内使用，明确不做大、不商业化、不规模化。**
因此本波所有设计以"简单、够用、防止滑向群发"为原则，不引入任何为规模准备的复杂度。

---

## 1. 本波目标（就两件事）

1. **发给最多 3 个联系人**：一次把同一条文本发给 1-3 个已选联系人（硬上限 3）。
2. **发到群**：能列出"我加入的群"，选一个群，发文本到群里。

**不做**：发给超过 3 人、群发到多个群、多账号、媒体消息、收群消息的复杂处理（本波群只做"发"，收可选）、发信体验的精细优化。

---

## 2. 为什么要硬上限 3（必须实现，不是建议）

"发给 1 个人"和"发给很多人"在代码上是同一个循环，只是次数不同。一旦支持多人发送，
这个工具在技术形态上就接近群发工具——而群发正是高封号风险行为。

为守住"小圈子兴趣"的定位、保护账号，**必须**：
- 在 **wrapper 层**（不只是 UI）硬编码"单次发送目标 ≤ 3"，超过直接拒绝返回错误。
- 保留第二波已有的发送节流（每条之间随机间隔）和 risk-stop 机制。
- 这个上限是账号安全的保险，不允许通过配置或 UI 绕过。

---

## 3. 功能一：发给最多 3 个联系人

### Go wrapper
- 复用现有发送逻辑。新增或改造发送入口，接受**多个目标 JID**（数组/JSON）。
- 内部循环逐个发送，每个之间保留随机节流间隔。
- **硬校验：目标数量 > 3 时，整体拒绝，返回明确错误（如 "exceeds max 3 recipients"）。**
- 逐个记录每个收件人的发送结果（成功/失败 + server_msg_id），整体返回结果列表 JSON。

```go
// 发送给 1-3 个联系人。toJidsJson 是 JSON 数组字符串，如 ["jid1","jid2"]。
// 内部校验数量 ≤ 3，否则拒绝。逐个发送（带节流），返回每个的结果 JSON。
SendTextMulti(toJidsJson string, text string, clientMsgId string) (string, error)
```

### Android UI
- 联系人列表改为**多选**（checkbox），但**最多只能选 3 个**——选到第 3 个后，其余置灰或提示"最多 3 人"。
- 选完 + 输入文本 → 发送 → 显示每个收件人的发送结果。
- UI 的 3 人限制和 wrapper 的硬上限是**双层保险**，两层都要有。

---

## 4. 功能二：发到群

### Go wrapper
- 新增获取群列表的方法。whatsmeow 有获取已加入群的能力（如 GetJoinedGroups）。
- 返回 JSON 列表 `[{"jid":"xxx@g.us","name":"群名","participant_count":N}]`。
- 发送复用现有逻辑——群 JID（@g.us）传给同一个发送方法即可，whatsmeow 内部不区分。

```go
// 获取当前账号已加入的群列表，返回 JSON。
GetGroups() (string, error)
```

### Android UI
- 加一个"群列表"入口/标签页，展示 GetGroups 结果。
- 选一个群 → 输入文本 → 发送到群。
- **注意**：发群是"发到 1 个群"，不是"发到多个群"。本波不支持一次发多个群（那是群发变体，禁止）。

### 收群消息（本波可选，建议先不做）
- 如果实现收群消息：收到的群消息事件要能区分"哪个群 + 群里哪个成员发的"，比 1对1 多一层信息。
- **建议本波先不做收群消息**——群消息量大、首次登录历史洪流更猛，先把"发群"做稳。收群留到下一波按需做。

---

## 5. 构建顺序

- **Phase 1**：wrapper 加 `SendTextMulti`（含 ≤3 硬校验）+ 桌面或单元测试验证"发 2-3 人成功、发 4 人被拒"。
- **Phase 2**：wrapper 加 `GetGroups` + 验证能列出群、能发到群。
- **Phase 3**：Android UI——联系人多选（≤3）+ 群列表 + 发送结果展示。
- **Phase 4**：真机验收 + trace 覆盖多人/群发送。

每个 Phase 完成停下来汇报，对应 ACCEPTANCE_WAVE3 的条目。

## 7. 安全红线（延续，不可破）
- 单账号、文本消息。
- 发给个人：**硬上限 3 人**。发群：**一次 1 个群**。
- 不做群发到多群、不做多账号、不做媒体、不做风控规避。
- 保留节流 + risk-stop。
- trace 不记 session key / 完整号码 / 消息正文（UI 实时回调可含正文）。
