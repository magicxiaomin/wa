# ACCEPTANCE · PoC 验收 checklist

每条都要可观察、可判定。PoC"通过"= 下面硬指标全绿。

## 硬指标（必须全部通过，否则 PoC 不算通过）
- [ ] 程序能 `go run` 启动，输出 `bridge_started` 事件
- [ ] 全新状态下能生成二维码（终端可见 / 或输出二维码内容），事件 `qr_generated`
- [ ] 用测试小号扫码后，进入 `connected` 状态
- [ ] 向白名单号码发送 1 条文本，收到 `message_sent` 且 payload 含服务器 message ID
- [ ] 向**不在白名单**的号码发送时，被拒绝并报错（验证防滥用约束生效）
- [ ] 超过发送计数上限时，被拒绝（验证计数约束生效）
- [ ] **关闭程序后重新运行，不弹二维码，直接进入 `connected`（session 恢复）**，事件 `session_restored`
- [ ] 能导出 `trace.json`，含从启动到发送的完整事件序列
- [ ] `trace.json` 中**不包含** session key / 消息正文明文 / 完整号码（脱敏生效）
- [ ] 发生 Go panic 时被 recover，转成 `error` 事件，进程不崩溃
- [ ] `ClearSession()` 后重新运行，重新需要扫码（验证清除生效）

## 尽力指标（拿到更好，拿不到不阻塞通过）
- [ ] 能记录到 `message_ack`（送达/已读回执）
- [ ] 断网后重连能恢复 `connected`，trace 里可见重连过程

## 项目级判定（这才是 PoC 的真正目的）
跑完上面之后，回答这三个问题，决定项目是否继续：
1. **能连上吗**：测试号能稳定登录、保持连接吗？
2. **能发出吗**：消息能稳定发到服务器（拿到 message ID）吗？
3. **会被封吗**：用一段时间后，测试号是否被封 / 限制？

- 三个都 OK → 项目地基成立，进入第二波（Android 交接）。
- 任一不 OK → 停下，重新评估这条非官方路线是否可行，不要进入 Android 工程。
