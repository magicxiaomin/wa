# 00_第二波如何使用（给你自己看）

## 这是什么
第二波交接包：把已验证的 whatsmeow 核心编进 Android，实现**获取联系人 + 收发指定消息**。
本波**不做发信体验优化**，只打通通路。

## 前置
- 第一波桌面 PoC 已通过（连接/发送/session 恢复都验证过）。
- 你选择了走法 B：接受封号风险无法提前完全确认，继续推进，边做边用测试小号观察。

## 文件清单
- `SPEC_ANDROID.md` — 主指令，6 个 Phase（A→F）的构建顺序
- `API_CONTRACT_ANDROID.md` — wrapper 新增方法/事件 + Kotlin 对接
- `ANDROID_PITFALLS.md` — Android/gomobile 特有坑（逐条对照）
- `ACCEPTANCE_ANDROID.md` — 分 Phase 验收
- 第一波的 KNOWN_PITFALLS / GOMOBILE_CONSTRAINTS / TRACE_SCHEMA 仍有效

## 怎么交给 Codex
把这些文件加到仓库（建议放仓库根，或一个 android/ 子目录），告诉 Codex 第一波已通过、现在做第二波，用下面的启动 prompt。

## 你要盯的关键点
1. **守住 Phase A 先在桌面做**：别让 Codex 一上来就进 Android。收消息/联系人功能先在桌面验证，再编进手机。这是最重要的纪律。
2. **每个 Phase 停下来验收**：尤其 Phase B（AAR 能不能编出来、能不能加载）是最可能卡时间的工程关，过了再往下。
3. **跨线程+跨进程**那个双重边界（ANDROID_PITFALLS 第1条），是收消息能不能正确到 UI 的命门，让 Codex 一开始就设计对。
4. 准备好测试小号 + 一个会跟你互发消息的联系人号（验收 Phase E 要用）。

## 本波之后
通路打通后，第三波才是发信体验优化（你最初的产品初衷）。到时再做交接。

# 可直接粘给 Codex 的启动 prompt

```
第一波桌面 Go PoC 已验证通过：whatsmeow 能连接、能发消息、session 能恢复。
现在进入第二波：Android 集成。请先完整阅读仓库里这些文档：
SPEC_ANDROID.md, API_CONTRACT_ANDROID.md, ANDROID_PITFALLS.md, ACCEPTANCE_ANDROID.md，
以及第一波仍然有效的 KNOWN_PITFALLS.md, GOMOBILE_CONSTRAINTS.md, TRACE_SCHEMA.md。

本波目标：把已验证的 whatsmeow wrapper 用 gomobile 编进 Android，实现
获取联系人 + 收发指定联系人的 1 对 1 文本消息。

重要范围限制：
- 本波不做发信体验优化（UI 能用即可，不追求好用）—— 那是第三波。
- 不做群发/多账号/群组/媒体消息。
- 这次要"收"消息，所以引擎放独立进程 :wa_bridge + ForegroundService 常驻（这是必要的）。

严格按 SPEC_ANDROID.md 第4节的 Phase A→B→C→D→E→F 顺序推进：
- Phase A 必须先在桌面完成（给 wrapper 加收消息+联系人并验证），再进 Phase B 编 Android。
  不要在 Android 环境里同时调试协议逻辑和集成问题。
- 每个 Phase 完成后停下来，说明实现了什么、如何验证、对应 ACCEPTANCE_ANDROID 哪几条，
  等我确认后再进下一个 Phase。

特别注意（来自坑清单）：
- 回调跨进程+跨线程双重边界，UI 更新必须切主线程（ANDROID_PITFALLS 第1条）
- gomobile 类型限制：复杂数据一律 JSON，whatsmeow 内部类型不外泄
- 优先 modernc.org/sqlite 纯 Go 驱动避开 cgo
- 前台服务 dataSync 类型 + POST_NOTIFICATIONS 权限（Android 13/14/15）
- 独立进程要用 getProcessName 区分初始化

先从 Phase A 开始：在桌面给 wrapper 加 GetContacts() 和接收消息事件，并验证能收发联系人消息。
```
